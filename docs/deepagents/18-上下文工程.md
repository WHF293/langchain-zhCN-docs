# 第十八章 上下文工程

> 来源: [Deep Agents 官方文档](https://docs.langchain.com/deep-agents)

---

上下文工程（Context Engineering）是构建高效代理系统的核心支柱。它决定了代理在每个推理步骤中能够访问哪些信息，以及这些信息如何被组织和压缩。本章将深入探讨上下文的各个组成部分、压缩策略以及长期记忆机制。

## 上下文的组成

代理的上下文（Context）由多个层次的信息构成，每个层次在不同的时间点注入并发挥不同的作用。

### 输入上下文

输入上下文（Input Context）是在代理开始执行之前就已经确定的信息集合。

#### 系统提示

系统提示（System Prompt）定义了代理的基本行为、角色和约束条件。它是代理最持久的上下文组成部分。

```typescript
// 定义系统提示
const systemPrompt = `你是一个专业的代码审查助手。

## 角色定义
- 专注于 TypeScript 代码审查
- 遵循 Google TypeScript 风格指南
- 优先关注安全性和性能问题

## 行为约束
- 每次审查最多提出 5 个改进建议
- 对每个问题提供具体的代码示例
- 使用严重程度分级：严重、警告、建议

## 输出格式
- 使用 Markdown 格式
- 包含行号引用
- 提供修复建议代码块`;

// 将系统提示注入代理配置
const agent = createAgent({
  systemPrompt,
  // ...其他配置
});
```

> **注意**：系统提示应当简洁明了，避免冗长的指令。研究表明，过长的系统提示可能导致代理性能下降。

#### 记忆

记忆（Memory）是代理在对话过程中积累的上下文信息，包括用户偏好、历史交互和学习到的模式。

```typescript
// 配置记忆系统
const memoryConfig = {
  // 短期记忆：当前对话的上下文
  shortTerm: {
    maxTokens: 4000,
    strategy: "sliding_window",
  },
  // 长期记忆：跨对话的持久信息
  longTerm: {
    storage: "vector_store",
    retrievalTopK: 5,
  },
};

// 在代理中使用记忆
const agentWithMemory = createAgent({
  memory: memoryConfig,
  // ...其他配置
});
```

| 记忆类型 | 生命周期 | 存储位置 | 适用场景 |
|----------|----------|----------|----------|
| 短期记忆 | 当前对话 | 内存 | 对话上下文、临时状态 |
| 工作记忆 | 当前任务 | 内存 | 任务进度、中间结果 |
| 长期记忆 | 持久化 | 数据库 | 用户偏好、学习模式 |

#### 技能

技能（Skills）是代理可以调用的预定义能力模块，它们为代理提供特定领域的专业知识和操作能力。

```typescript
// 定义技能
const codeReviewSkill = {
  name: "code_review",
  description: "执行代码审查并生成报告",
  instructions: `审查步骤：
1. 检查代码风格一致性
2. 识别潜在的安全漏洞
3. 评估性能影响
4. 验证错误处理
5. 生成审查报告`,
  tools: ["ast_parser", "security_scanner", "style_checker"],
};

// 将技能注册到代理
const agent = createAgent({
  skills: [codeReviewSkill],
  // ...其他配置
});
```

#### 工具提示

工具提示（Tool Prompts）为代理提供关于如何使用特定工具的详细指导。

```typescript
// 定义工具及其提示
const searchTool = {
  name: "web_search",
  description: "在互联网上搜索信息",
  prompt: `使用此工具时：
- 使用精确的关键词，避免过于宽泛的查询
- 对于技术问题，包含版本号或日期限定
- 验证多个来源的一致性
- 注意信息的时效性`,
  parameters: {
    query: { type: "string", required: true },
    maxResults: { type: "number", default: 10 },
  },
};
```

### 运行时上下文

运行时上下文（Runtime Context）是在代理执行过程中动态生成和更新的信息。

```typescript
// 运行时上下文结构
interface RuntimeContext {
  // 当前对话轮次
  turnNumber: number;
  // 用户最近的输入
  latestUserInput: string;
  // 工具调用结果
  toolResults: ToolResult[];
  // 代理的中间推理状态
  reasoningState: ReasoningState;
  // 时间戳
  timestamp: Date;
  // 环境信息
  environment: {
    userAgent: string;
    locale: string;
    timezone: string;
  };
}

// 在代理执行过程中更新运行时上下文
function updateRuntimeContext(
  context: RuntimeContext,
  event: AgentEvent
): RuntimeContext {
  return {
    ...context,
    turnNumber: context.turnNumber + 1,
    timestamp: new Date(),
    // 根据事件类型更新相应字段
  };
}
```

> **提示**：运行时上下文应当与输入上下文明确分离，这样可以更灵活地管理不同类型的信息。

## 上下文压缩

随着对话的进行，上下文会不断增长。上下文压缩（Context Compression）是管理上下文大小的关键技术。

### 卸载

卸载（Offloading）是将不活跃的上下文信息转移到外部存储的技术。

```typescript
// 实现上下文卸载
class ContextOffloader {
  private archiveStore: VectorStore;

  // 卸载旧的对话轮次
  async offloadTurns(
    turns: ConversationTurn[],
    maxActiveTurns: number
  ): Promise<OffloadResult> {
    if (turns.length <= maxActiveTurns) {
      return { offloaded: [], active: turns };
    }

    // 将旧轮次存档到向量存储
    const toOffload = turns.slice(0, turns.length - maxActiveTurns);
    const active = turns.slice(-maxActiveTurns);

    await this.archiveStore.addDocuments(
      toOffload.map((turn) => ({
        pageContent: this.serializeTurn(turn),
        metadata: { turnId: turn.id, timestamp: turn.timestamp },
      }))
    );

    return { offloaded: toOffload, active };
  }

  // 根据相关性检索卸载的内容
  async retrieveRelevant(
    query: string,
    topK: number = 3
  ): Promise<Document[]> {
    return this.archiveStore.similaritySearch(query, topK);
  }
}
```

| 卸载策略 | 触发条件 | 优点 | 缺点 |
|----------|----------|------|------|
| 滑动窗口 | 超过固定轮数 | 实现简单 | 可能丢失重要信息 |
| 基于时间 | 超过时间阈值 | 符合自然遗忘 | 需要时间戳管理 |
| 基于相关性 | 语义距离超过阈值 | 保留相关信息 | 计算开销较大 |
| 混合策略 | 多条件组合 | 平衡各因素 | 配置复杂 |

### 摘要

摘要（Summarization）是将冗长的上下文压缩为更简洁形式的技术。

```typescript
// 实现上下文摘要
class ContextSummarizer {
  private llm: BaseLanguageModel;

  // 生成对话摘要
  async summarizeConversation(
    turns: ConversationTurn[]
  ): Promise<ConversationSummary> {
    const conversationText = turns
      .map((t) => `${t.role}: ${t.content}`)
      .join("\n");

    const summaryPrompt = `请将以下对话压缩为简洁的摘要，保留关键信息：

${conversationText}

摘要要求：
1. 保留主要话题和决策
2. 记录用户的重要偏好
3. 包含待解决的问题
4. 长度不超过原文的 20%`;

    const summary = await this.llm.invoke(summaryPrompt);

    return {
      text: summary,
      originalTurns: turns.length,
      compressedTokens: this.countTokens(summary),
      timestamp: new Date(),
    };
  }

  // 增量摘要：将新内容合并到现有摘要
  async incrementalSummarize(
    existingSummary: string,
    newTurns: ConversationTurn[]
  ): Promise<string> {
    const newContent = newTurns
      .map((t) => `${t.role}: ${t.content}`)
      .join("\n");

    const prompt = `现有摘要：
${existingSummary}

新的对话内容：
${newContent}

请将新内容合并到现有摘要中，保持简洁并更新相关信息。`;

    return this.llm.invoke(prompt);
  }
}
```

> **建议**：对于关键信息（如用户明确表达的偏好），应当保留在显式字段中，而不是仅依赖摘要。

## 子代理的上下文隔离

在多代理系统中，子代理（Sub-Agent）需要与其父代理和其他子代理保持适当的上下文隔离。

```typescript
// 实现子代理上下文隔离
class SubAgentContextManager {
  // 为主代理创建隔离的子代理上下文
  createIsolatedContext(
    parentContext: AgentContext,
    task: Task
  ): SubAgentContext {
    return {
      // 继承必要的全局信息
      systemPrompt: this.buildSubAgentPrompt(task),
      // 不继承父代理的对话历史
      conversationHistory: [],
      // 只传递任务相关的工具
      tools: this.filterToolsForTask(parentContext.tools, task),
      // 创建新的记忆空间
      memory: new IsolatedMemory(),
      // 添加任务特定的上下文
      taskContext: {
        description: task.description,
        constraints: task.constraints,
        expectedOutput: task.expectedFormat,
      },
    };
  }

  // 合并子代理结果到父上下文
  mergeSubAgentResult(
    parentContext: AgentContext,
    subAgentResult: SubAgentResult
  ): AgentContext {
    return {
      ...parentContext,
      // 将子代理结果添加到父代理的上下文
      toolResults: [
        ...parentContext.toolResults,
        {
          tool: "sub_agent",
          result: subAgentResult.output,
          metadata: {
            subAgentId: subAgentResult.agentId,
            executionTime: subAgentResult.duration,
          },
        },
      ],
    };
  }
}
```

> **注意**：上下文隔离可以防止子代理访问不应有的信息，同时避免上下文污染（Context Contamination）。

## 长期记忆

长期记忆（Long-Term Memory）使代理能够在多次对话之间保留和检索信息。

```typescript
// 实现长期记忆系统
class LongTermMemory {
  private store: VectorStore;
  private graph: KnowledgeGraph;

  // 存储重要信息
  async remember(key: string, value: any, metadata: MemoryMetadata): Promise<void> {
    // 存入向量存储用于语义检索
    await this.store.addDocuments([{
      pageContent: JSON.stringify({ key, value }),
      metadata: {
        ...metadata,
        timestamp: new Date(),
        accessCount: 0,
      },
    }]);

    // 更新知识图谱
    await this.graph.addNode(key, value, metadata.relations);
  }

  // 基于语义相似性检索记忆
  async recall(query: string, options: RecallOptions = {}): Promise<Memory[]> {
    const {
      topK = 5,
      minRelevance = 0.7,
      timeDecay = true,
    } = options;

    let results = await this.store.similaritySearchWithScore(query, topK * 2);

    // 应用时间衰减
    if (timeDecay) {
      results = results.map(([doc, score]) => {
        const age = Date.now() - new Date(doc.metadata.timestamp).getTime();
        const decayFactor = Math.exp(-age / (30 * 24 * 60 * 60 * 1000)); // 30天衰减
        return [doc, score * decayFactor] as [Document, number];
      });
    }

    // 过滤低相关性结果
    return results
      .filter(([_, score]) => score >= minRelevance)
      .slice(0, topK)
      .map(([doc, score]) => ({
        content: JSON.parse(doc.pageContent),
        relevance: score,
        metadata: doc.metadata,
      }));
  }

  // 基于知识图谱的关系检索
  async recallByRelation(
    entity: string,
    relation: string,
    depth: number = 2
  ): Promise<Memory[]> {
    const relatedNodes = await this.graph.traverse(entity, relation, depth);
    return relatedNodes.map((node) => ({
      content: node.data,
      relevance: 1.0 / node.distance,
      metadata: node.metadata,
    }));
  }
}

// 使用长期记忆的代理配置
const agentWithLongTermMemory = createAgent({
  longTermMemory: new LongTermMemory({
    store: new ChromaVectorStore(),
    graph: new Neo4jKnowledgeGraph(),
  }),
  // ...其他配置
});
```

| 记忆检索策略 | 适用场景 | 优点 | 缺点 |
|--------------|----------|------|------|
| 语义相似性 | 自然语言查询 | 灵活、直观 | 可能遗漏精确匹配 |
| 关键词匹配 | 精确查找 | 快速、准确 | 缺乏语义理解 |
| 知识图谱遍历 | 关系查询 | 捕捉复杂关系 | 需要预建图谱 |
| 混合检索 | 综合场景 | 全面覆盖 | 实现复杂 |

> **提示**：长期记忆系统应当实现自动遗忘机制，定期清理过时或不再相关的信息，以保持记忆库的质量。

## 上下文工程最佳实践

1. **分层管理**：将上下文分为系统级、会话级和任务级，分别管理
2. **按需加载**：只在需要时才将信息注入上下文，避免信息过载
3. **定期压缩**：实施主动的上下文压缩策略，而不是等到超出限制
4. **隔离与共享**：在多代理系统中，明确哪些信息需要隔离，哪些需要共享
5. **监控与调优**：持续监控上下文使用情况，根据实际效果调整策略

> **建议**：在生产环境中，建议实现上下文使用指标（Context Usage Metrics），包括上下文大小、压缩率、检索命中率等，以便持续优化上下文工程策略。
